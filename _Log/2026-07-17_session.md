# Session Log — 2026-07-17

## Summary

이번 세션의 성격이 이전과 다르다: **JY가 명시적으로 "내가 풀거야, 절대 풀이나 답을 알려주지마"**라고 선을 그었고(→ 메모리 `feedback-no-solutions`), 그에 따라 나는 **완성된 풀이를 주지 않고 building block(문법·개념·why)과 Socratic 질문만** 제공했다. JY가 두 레벨을 **자력으로** 뚫었고, 노트는 그가 실제로 밟은 경로를 사후 문서화한 것.

두 레벨의 축:

1. **L24 (24→25, ★★☆)** — **online brute-force + connection batching**. `localhost:30002` 데몬에 `<pw> <PIN>` 만 줄을 던진다. JY가 `{0000..9999}` brace expansion으로 파일 생성 → `nc localhost 30002 < all`(단일 연결) → `sort | uniq -u`로 정답 분리. 진짜 교훈은 "매 추측 재연결 vs 단일 연결 재사용"의 비용 차 = **병목은 계산이 아니라 TCP handshake**.
2. **L25 (25→26, ★★★)** — **restricted shell escape**. bandit26 로그인 셸이 `/usr/bin/showtext`(=`exec more ~/text.txt`)인 감옥. pager→editor→shell 체인(`more`에서 `v` → vi → `:set shell=/bin/bash` → `:sh`)으로 탈출. 함정의 정체 = **`$SHELL` 상속**: `more`의 `!`·vi의 `:sh`가 오염된 `$SHELL`(=showtext)을 참조해 감옥으로 튕김. 돌파는 vi의 **재설정 가능한 `shell` 옵션**.

**하이라이트 — L25의 `$SHELL` 상속 트랩.** JY가 `man more`에서 `!command`(subshell)를 찾아 `!cat …`을 쳤으나 명령이 씹히고 ASCII 아트만 재출력 → "왜 안 먹지"에서 막힘. 진단: `more`의 `!`는 `$SHELL -c`를 실행하는데 `$SHELL=/usr/bin/showtext`라 `exec more`로 복귀(fixed point). `man more`의 `v`(editor는 `$EDITOR`/`$VISUAL` 경유, `$SHELL` 우회)로 전환 → vi 진입 → 그런데 `:sh`도 같은 `$SHELL` 트랩. 최종 열쇠: `:set shell?`로 오염값 확인 → `:set shell=/bin/bash` 재할당(환경변수가 아닌 **옵션**이라 가능) → `:sh` → bandit26 셸. **"오염은 변수로 전파, 돌파는 사용자-가변 옵션으로."**

**보안**: 노출 credential = bandit24 password(brute-force 생성 명령의 `echo` 인자에 리터럴), bandit25 password(Correct! 응답), bandit26 password(`cat /etc/bandit_pass`), 그리고 **bandit26 OpenSSH private key 전문**. 전부 노트에서 `<… masked>` 처리, **private key는 본문에서 완전 제외**(파일명으로만 참조 — PEM 헤더 리터럴도 미기재해 스캐너 오탐 차단). 커밋 전 grep 전수 + pre-commit `--dry-run` 필요.

## Struggle Points (paste에서 관측)

- **L24 — 파이프 미연결 버그**: 첫 스크립트가 `nc localhost 30002`를 루프 **위에 단독**으로 둠 → nc가 자기 stdin(터미널)을 기다리고 루프 `echo`는 nc 종료 후 터미널로. 둘이 파이프로 안 이어짐(순차 실행). `nc < all`(명시적 redirect)로 해결. **"인접 배치 ≠ 연결."**
- **L24 — zero-pad / uniq 취약성**: `{0000..9999}`의 leading-zero pad를 활용(good). 다만 `sort|uniq -u`는 "1회 등장"으로 거른 운 좋은 필터 — `grep -v Wrong`이 더 견고함을 Phase 4에 기록.
- **L25 — resize 타이밍(막힘 1)**: "터미널 아무리 줄여도 close된다". 원인은 **순서** — `more`는 시작 시 pty 크기를 한 번 읽으므로 **접속 후** 줄이면 늦음. 접속 **전** 창 축소가 정답. 부수 오해: "text.txt가 1줄"(실은 접속 MOTD를 text.txt로 착각; 실제론 여러 줄 아트).
- **L25 — `$SHELL` 상속 트랩(막힘 2, 핵심)**: `more`의 `!cat …`·vi의 `:sh`가 전부 무력(명령 씹히고 아트 재출력). 오염된 `$SHELL`(showtext) 참조가 원인. `v`(→$EDITOR)로 1차 우회 + vi `shell` **옵션 재설정**으로 2차 우회.
- **L25 — 원격 vim 클립보드(곁길)**: `y`(yank)가 ⌘ 클립보드에 안 감 — vim register ≠ OS clipboard(+clipboard·$DISPLAY 필요, SSH 너머라 서버 클립보드일 뿐). 터미널 드래그도 vim `mouse=a`가 가로챔 → `:set mouse=` 또는 Option-드래그로 해결.
- **워크플로 — 자력 풀이 제약**: JY 요청으로 완성 풀이를 주지 않고 building block만 제공. Socratic 질문("more가 안 죽는 조건?", "`$SHELL` 안 거치는 실행 경로?")으로 문 앞까지만 안내. → 메모리 `feedback-no-solutions` 신설.
