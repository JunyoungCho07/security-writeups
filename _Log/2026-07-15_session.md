- **L15 — openssl 노이즈 = "뽀록" 착시**: handshake 진단(cert/session-ticket/`read R BLOCK`)이 화면을 덮어 "왜 됐는지 모름". 실체는 "TLS 입은 nc" + `verify error`는 신원 문제지 암호화 실패 아님. `-quiet`로 노이즈 걷으면 본질 노출.

### Summary

L20에서 예고된 **cron 3부작**을 연속 클리어. 세 레벨이 같은 `/etc/cron.d/` 무대에서 난이도를 계단식으로 올린다:

1. **L21** (★★☆) — `cronjob_bandit22.sh`가 **고정 경로** `/tmp/<name>`에 bandit22 pw를 `chmod 644`로 덤프 → 그냥 `cat`. cron **USER 필드 = 권한 컨텍스트**(setuid의 평행) 첫 등장.
2. **L22** (★★☆) — 파일명이 `md5("I am user $(whoami)")`로 **파생**. cron이 bandit23으로 도니 그 해시를 **재현**해 `/tmp/<hash>` read. 스크립트 직접 실행 함정(whoami=bandit22 → 자기 pw). MD5 결정성.
3. **L23** (★★★, ~45min) — **confused deputy / script injection**. `/var/spool/bandit24/foo`에 bandit23-소유 실행 스크립트를 심으면 cron이 **bandit24로 대신 실행** → 그 스크립트가 `/etc/bandit_pass/bandit24`를 읽어 world-writable dir로 흘림. L21·22가 "cron 산출물 read"였다면 L23은 **"cron이 실행할 코드를 주입"** — 읽기가 아니라 실행 흐름 하이재킹.

**하이라이트 — L23은 논리보다 디버깅이 벽이었다.** exploit 구조는 금방 이해했으나 cron이 stdout/stderr를 `&>/dev/null`로 삼켜 **모든 실패가 무증상**("스크립트는 사라졌는데 결과 없음"). 뚫은 3개 열쇠: ① **printf로 스크립트 생성**(에디터/붙여넣기 CRLF `\r` → `/bin/bash\r bad interpreter` 제거) ② **`chmod 777` on `mktemp -d`**(기본 700이라 bandit24 write 불가) ③ **`id > $D/proof` 자가 로그**(실행됐나 + 누구로? → `uid=11024(bandit24)` 확정). 최종 클리어는 JY 본인 스타일(heredoc + 최소 payload)로 재현 → 노트 Solution도 그 스타일로 확정.
